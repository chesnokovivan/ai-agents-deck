MCP
стандарт подключения агенту источников данных и инструментов.

Шесть форм, под каждой — что я строил

RAG / Q&A над знаниями
«спроси у доков» — retrieval → контекст → ответ со ссылками.Грабли: качество = качество поиска; искать или нет решает описание KB.
NL → SQL аналитика
вопрос словами → SQL → хранилище → бизнес-инсайт. Мой главный кейс.Грабли: синтаксически верный, но семантически неправильный SQL.
Data-in → рекомендация
симуляторы продаж: один LLM — персона, второй судит по рубрике.Это и «анализ по маске», и LLM-as-judge.
Агент-исполнитель
триаж в Slack: эскалация → поиск в 3 источниках → вердикт в треде.Опасные действия — за PreToolUse hook, подтверждение в 1 клик.

7 исследователей → дайджест → 6 скептиков → синтез.Почти не писал код — гонял workflow. → демо.

Слой 1 — Ingestion

Подключение к данным (per-source)

У тебя слот для Presto / Redash / Tableau / Snowplow — каждый встаёт как MCP-сервер за тем же агентом.

Обучаемость

Агент не дообучается — он накапливает память

Петля
read-before-query, write-after-discovery. Перед запросом читает накопленные gotchas; после открытия — записывает.
Маршрутизация
таблица в промпте «тип открытия → файл»: новая колонка → схема, странность → gotchas, удачный запрос → patterns.
Critical Overrides
наверх вынесены ценные исправления: «метрики в ml.*», «churn_treshold с опечаткой — пиши так».
Источник + свежесть
на каждом файле — источник и предупреждение: память ≠ live-состояние.
Из практики: агент выучил схему данных компании, прочитав Slack-канал другого бота — реальные имена таблиц и формулы метрик. Знание, которое ему никто не давал.

✓
данные как контракт: схема + определения метрик задокументированы

✓
dry-run + потолок стоимости (критично для Presto — pay-per-scan)
✓
слой памяти: routing + Critical Overrides + источник
✓
golden/regression-набор вопросов с первого дня
✓
промпты под версионированием, логи решений агента
✓
границы прав на любое необратимое действие (hook-гейт)

Контексты и промпты

  • LLM рассказывает, код считаетжёсткие числа — из SQL/сервиса, не из модели (LLM галлюцинировал 30% речи там, где было 0%)
  • Семантический слой инлайн бьёт vector-retrievalнаивный SUM завышал ×4: 2 015 → 8 060. Модель ошибётся, если не сказать
  • Никаких выдуманных метриктолько инструментированные числа; выдуманное число подрывает доверие ко всему
  • Канонические данные — первымиспроси команду, какую таблицу считают канонической, не предполагай
  • Стоимость — safety rail с самого началаdry-run обязателен, $-порог в промпте

Гибрид: тонкий Slack-листенер + управляемый мозг

Схема ниже = вариант B (мозг в облаке Anthropic + туннель к данным). Прод-базу (вариант A) и трейдоффы — см. баннер.

⚠️ Данные за VPN — ни один облачный вариант не держит их буквально в периметре. Выбор по тому, КУДА уходят данные и кто их хранит:
✓ A · Agent SDK в кластере + Bedrock/Vertex (прод-база) — наружу только вызов модели; данные остаются в сети/регионе провайдера (PrivateLink/PSC), не у Anthropic, без retention. B · Managed Agents + MCP-туннель — мозг в облаке, исходящий gateway к данным. Быстрый PoC. ⚠️ stateful (не ZDR/HIPAA), гео US, beta. C · Managed Agents + self-hosted sandbox — исполнение в твоей инфре, но tool-вызовы (вкл. результаты SQL) всё равно идут на control-plane Anthropic — не air-gapped.
Реальный блокер — не «достать данные» (туннель/sandbox решают это), а retention / data-residency / PII. Полный разбор → oleg_deployment_options.md

  • Managed Agents — Agent + Sessions
  • кастомный skill (аналитика)
  • MCP: Presto / Redash / Metabase — read-only (нужен доступ из периметра — см. выше)

Путь запроса
  1. Вопрос в Slack-канале аналитики
  2. Событие message.channels → листенер по Socket Mode
  3. Листенер: POST /v1/sessions, ключ = thread_ts, цепляет Memory + Vaults
  4. Шлёт user.message с вопросом
  5. Skill: уточнение → сбор данных через MCP (NL→SQL) → ответ с источниками
  6. Ответ стримится по SSE → листенер постит в тред
  7. Опасное действие — PreToolUse hook + аппрув в треде
  8. Контекст и память живут в Memory Store между ходами
Почему гибрид: Managed Agents работает только по вызову — своего Slack-триггера у него нет, поэтому нужен тонкий листенер. Инфру держать не хотим → мозг отдаём в managed. Креды к БД остаются в Vaults, листенер их не видит.
